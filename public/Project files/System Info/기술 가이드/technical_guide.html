주요 기술 가이드

Hansoll AI Portal을 구동하는 핵심 기술 스택 및 구조 안내

프론트엔드 (Frontend)

  • Tailwind CSS:

    유틸리티 기반의 CSS 프레임워크를 사용하여 빠르고 일관된 반응형 디자인 및 Glassmorphism 효과를 구현했습니다.

  • Vanilla JS & React (CDN):

    가벼운 화면 렌더링은 순수 자바스크립트를, 복잡한 상태관리나 애니메이션(파티클 등)에는 React CDN 방식을 채택하여 유연성을 높였습니다.

백엔드 및 시스템 (Backend)

  • Node.js & Express:

    로컬 환경에서 빠르고 가벼운 API 서버를 구동하며, 파일 업로드와 JSON 데이터 읽기/쓰기를 처리합니다.

  • JSON 기반 로컬 DB:

    복잡한 RDBMS 대신 가벼운 JSON 파일을 데이터베이스로 활용하여 메뉴 설정 및 카테고리 상태를 영구 저장합니다.

인터랙션 및 애니메이션 (Animations)

HTML5 Canvas 물리 엔진

브랜드 트랜지션에 사용된 파티클(점) 애니메이션은 Canvas 2D API와 자체 물리 엔진(Spring & Friction)을 결합하여 구현했습니다. 이미지를 픽셀 단위로 분석하여 수백 개의 파티클을 부드럽게 재구성합니다.

Cross-Origin 안전 통신

iframe 간의 통신 시 발생하는 CORS 에러를 우회하기 위해 URL Search Parameters와 postMessage API를 조합하여 로컬 환경(file://)에서도 완벽하게 동작하는 보안 구조를 구축했습니다.

서버리스 클라우드 DB (Supabase)

도입 배경 (로컬 저장소의 한계 극복)

초기에는 메뉴 데이터를 서버의 로컬 파일 시스템에 의존했으나, Vercel 클라우드 환경 특유의 읽기 전용(Read-Only) 정책으로 인해 관리자 페이지에서 변경한 데이터가 반영되지 않는 문제가 있었습니다. 이를 해결하고 '단 한 번의 배포도 필요 없는 실시간 동기화'를 달성하기 위해 강력한 서버리스 DB인 Supabase를 도입했습니다.

Project (프로젝트)

포털 서비스 전체를 감싸는 최상위 컨테이너입니다. 독립적인 데이터베이스와 클라우드 스토리지(Vercel Blob과 유사)를 한 곳에서 관리하며, 통신을 위한 고유한 접속 키(API Key)를 발급하여 보안을 유지합니다.

Database (데이터베이스)

대규모 트래픽을 견디는 PostgreSQL 기반의 관계형 데이터베이스입니다. 포털을 구동하는 모든 메타데이터와 시스템 설정값들이 영구적으로 보존되는 거대한 안전 금고의 역할을 수행합니다.

Table (테이블)

데이터베이스 내부의 실질적인 엑셀 시트(Sheet)와 같은 역할을 합니다. '메뉴 설정(MenuSetting)', '매뉴얼(MenualPage)' 등 데이터의 목적에 맞게 테이블을 분리하여 효율적인 추가, 수정, 조회를 가능하게 합니다.

클라우드 파일 저장소 (Supabase Storage)

도입 배경 (로컬 파일 동기화의 한계 극복)

Supabase로 텍스트 형식의 '데이터' 문제는 해결했으나, 신규 메뉴의 HTML 파일이나 파이썬 스크립트 등 물리적인 '파일 데이터'를 새로 추가하거나 코드를 변경할 때에는 여전히 로컬에 저장 후 전체 배포를 다시 해야만 Vercel에 반영되는 한계가 남아 있었습니다. 이를 완벽한 서버리스로 탈바꿈하기 위해 Supabase Storage를 도입했습니다.

핵심 기술 로직 (전면 클라우드 이관)

기존 하드디스크의 Project files/ 폴더 내부에 위치하던 모든 로컬 파일을 Vercel 환경 내부의 클라우드 스토리지(Blob)로 통째로 이관시켰습니다. 이제 포털은 메뉴 클릭 시 로컬 PC 경로가 아닌 클라우드 스토리지 상의 고유 URL 주소를 다이렉트로 바라보고 파일을 불러옵니다.

도입 성과 (완전한 배포 제로화 달성)

이제 관리자 페이지에서 파일을 업로드하거나 소스 코드를 수정하면, 즉시 클라우드 저장소 공간에 직접 덮어쓰기가 진행됩니다. 결과적으로 DB 데이터뿐만 아니라 '파일 데이터' 관리에서도 배포 과정이 100% 생략되며, 로컬 작업 없이 클라우드 환경 자체만으로 유지/보수가 가능한 진정한 서버리스 시스템이 완성되었습니다.

테넌트 계정 (Microsoft Entra ID) 기반 사내 보안 로그인

도입 배경 (보안 강화 및 편의성 증대)

허가되지 않은 외부 인원의 접근을 원천 차단하고, 임직원들이 별도의 번거로운 회원가입 절차 없이 이미 사용 중인 회사 업무용 계정으로 안전하게 포털에 접속할 수 있도록 Single Sign-On (SSO) 환경을 구축했습니다.

핵심 기술 로직 (Supabase Auth & Graph API)

Supabase의 인증(Auth) 시스템을 통해 Microsoft 서버로 안전하게 권한을 넘긴 뒤, Microsoft Graph API를 호출합니다. 단순히 영문 이름뿐만 아니라 사내 정책상 한글 이름이 저장된 '사무실 위치(officeLocation)' 속성과 '부서명(department)' 데이터를 즉시 불러와 화면에 띄워줍니다.

도입 성과 (부서 맞춤형 스마트 권한 제어)

모든 사용자가 본인의 한글 이름과 부서를 명확히 확인할 수 있으며, Graph API로 추출한 부서 데이터를 기반으로 "IT부" 소속에게만 관리자 버튼(톱니바퀴)을 자동으로 노출시켜 타 부서 인원의 접근을 차단하는 동적 보안 로직이 완성되었습니다.

상세 시스템 동작 가이드 (비전문가용 쉽게 풀이)

1. 실행기(.bat)와 npm run dev의 역할

비유하자면: 자동차의 '스마트키'와 '엔진'의 관계입니다.

npm run dev (엔진): 포털 사이트가 껍데기뿐인 그림이 아니라 실제로 작동하도록 뒤에서 데이터 톱니바퀴를 돌려주는 엔진(백엔드 서버)입니다. 이 엔진이 켜져 있어야만 엑셀 파일을 업로드하고 메뉴 이름을 수정/저장할 수 있습니다.

실행기.bat (스마트키): 매번 까만 화면을 열어 복잡한 명령어를 치고 엔진을 켜는 것은 매우 불편합니다. .bat 파일은 더블클릭 한 번만 하면 1) 엔진(npm run dev)을 알아서 켜주고, 2) 곧바로 크롬 브라우저를 열어 포털 화면까지 띄워주는 똑똑한 원터치 자동화 로봇입니다.

2. DB가 없는데 데이터는 어떻게 저장될까? (JSON 방식)

비유하자면: 거대한 '물류 창고(DB 프로그램)'를 빌리는 대신, 책상 서랍의 '비밀 장부(JSON 파일)'에 글씨를 써넣는 것과 같습니다.

물리적 저장 공간 및 저장 형태 (JSON)

이 장부는 허공(메모리)에만 존재하는 것이 아니라, 포털이 설치된 하드디스크의 정확한 폴더 경로(.data/menuData.json)에 물리적인 파일로 존재합니다. (현재 기준 d:\0 Hansoll AI\Hansoll AI Portal TEST2\.data\menuData.json)
저장 형태: 일반 메모장(txt)이나 엑셀(csv)처럼 내용이 글자로 저장되는 텍스트 파일이지만, 컴퓨터와 사람이 모두 쉽게 구조를 파악할 수 있도록 중괄호 {} 와 대괄호 [] 규칙을 사용해 작성된 JSON(제이슨) 포맷의 파일입니다.

PC(서버) 재부팅 시의 영향

우리가 엑셀이나 메모장(.txt) 파일을 D드라이브에 저장하고 컴퓨터를 껐다 켜도 파일이 날아가지 않듯, 이 방식도 완벽히 동일합니다. 데이터가 하드디스크 경로에 영구적인 텍스트 파일 형태로 꾹꾹 눌러 쓰여 있기 때문에, PC나 서버가 예기치 않게 꺼지거나 재부팅되어도 기존에 세팅한 메뉴와 설정값들은 100% 안전하게 보존됩니다. 재부팅 후 엔진(npm run dev)을 다시 켜기만 하면, 엔진이 해당 경로의 .json 파일을 찾아 그대로 읽어들입니다.

데이터 유지를 위한 내부 작업

관리자가 화면에서 '저장' 버튼을 누르는 즉시, 엔진(Node.js 서버)은 내부적으로 '파일 덮어쓰기(File System Write)'라는 작업을 수행합니다. 장부를 조용히 열어서 지우개로 지우고 새 이름으로 고쳐 쓴 뒤 곧바로 하드디스크에 저장해버립니다. 복잡한 DB 프로그램(오라클, MySQL 등) 없이도, 단지 이 파일 하나만 잘 복사해두면 언제든 전체 시스템이 그대로 백업되는 엄청난 장점이 있습니다.

[
  { "title": "사업 3부", "status": "Y" },
  { "title": "HR부", "status": "N" }
]
▲ menuData.json 파일 (장부) 내부 모습

3. 메뉴 등록, 그리고 "사용/숨김" 토글의 저장 원리

카테고리/메뉴 신규 등록 시

새 메뉴를 만들고 엑셀/HTML 파일을 첨부하여 저장하면, 1) 첨부된 파일은 Project files 폴더 내부의 자기 자리에 안전하게 복사되어 저장되고, 2) 장부(JSON 파일) 맨 아랫줄에는 해당 메뉴의 이름, 파일명, 아이콘 정보가 새롭게 한 줄 추가됩니다.

사용 유무 토글 방식

장부 안에는 각 메뉴마다 "status"(상태)라는 이름표가 붙어 있습니다. 스위치(토글)를 켜면 이 이름표가 "Y"(Yes)로 바뀌며 즉시 화면에 나타나고, 끄면 "N"(No)으로 바뀌며 화면에서만 쏙 숨겨집니다. 데이터를 삭제하는 것이 아니기 때문에 나중에 언제든 다시 살릴 수 있는 안전한 방식입니다.

4. 관리자 페이지 비밀번호는 어디에 저장되나?

비유하자면: 접속 중인 인터넷 브라우저(크롬, 엣지 등) 내부에 마련된 '나만의 개인용 금고'에 보관됩니다.

비밀번호는 앞서 말한 파일 장부(JSON)나 서버로 전송되지 않습니다. 브라우저가 기본적으로 제공하는 로컬 스토리지(Local Storage)라는 브라우저 자체 메모리 공간에 암호화되어 조용히 저장됩니다.

이 방식은 외부에서 파일 형태로 빼내거나 해킹할 위험이 원천 차단되며, 사용자의 PC 브라우저 안에서만 신원을 확인하므로 접속이 매우 빠르고 안전합니다.

5. 화면 상단에 나오는 '메뉴 건수'는 어떻게 계산될까?

계산 방식: '실시간 숫자 세기' 입니다.

어딘가에 "총 15개" 라고 고정된 숫자를 적어놓는 것이 아닙니다. 화면이 새로고침될 때마다 엔진이 장부(JSON)를 빠르게 처음부터 끝까지 훑어봅니다.

이때 숨김 처리된 메뉴("status"가 "N"인 것)는 제외하고, 현재 사용 중("status"가 "Y"인 것)인 카테고리와 하부 메뉴만 그 자리에서 하나, 둘, 셋 하고 직접 센 다음 화면에 "O건" 이라고 뿌려줍니다. 따라서 누군가 메뉴를 추가하거나 숨기면, 다음 순간 바로 계산기가 다시 돌아가 무조건 100% 정확한 숫자를 보여주는 똑똑한 방식입니다.

6. Vercel에 배포하면 왜 파이썬(Python) 프로그램이 실행되지 않을까?

비유하자면: Vercel은 그림을 걸어두는 '미술관'이지, 무거운 기계를 돌리는 '공장'이 아니기 때문입니다.

1) Vercel의 역할 (정적 호스팅)
Vercel은 HTML, CSS, JavaScript처럼 모니터 화면에 '보여주는' 문서들을 전 세계 사람들에게 아주 빠르게 전달하는 데 특화된 서비스입니다. 반면 파이썬(Python)은 단순히 화면을 그리는 것이 아니라, 컴퓨터 뒷단에서 스스로 연산을 하고, 엑셀 파일을 열어 데이터를 분석하거나 수정하는 등 무거운 '실제 작업'을 수행하는 프로그램입니다.

2) 로컬(내 컴퓨터)에서만 파이썬이 돌아가는 이유
여러분이 사용하시는 업무용 PC에는 파이썬(Python) 프로그램이 이미 설치되어 있습니다. 그래서 바탕화면의 실행기.bat 파일로 포털을 켜면, 포털이 내 컴퓨터 안에 있는 파이썬에게 "이 데이터 좀 처리해줘!" 하고 직접 명령을 내릴 수 있습니다.

3) 결론적으로
Vercel이라는 외국 서버에 포털을 배포하면, 사람들에게 화면(버튼)은 똑같이 보여줄 수 있습니다. 하지만 막상 파이썬 실행 버튼을 누르면, Vercel 서버 안에는 작업을 대신 해줄 '파이썬 직원'이 아예 존재하지 않으며 권한도 없기 때문에 작동하지 못하고 실패하게 됩니다. 따라서 CMPH Recap Tool 같이 파이썬 스크립트를 직접 가동하는 도구는 반드시 개인 PC(로컬 서버) 환경에서 실행기.bat 등을 통해 실행하셔야 정상적으로 작동합니다.

7. Vercel에서 관리자 페이지 '저장'이 적용되지 않는 이유 (로컬 파일 시스템의 한계)

비유하자면: Vercel은 '인쇄가 끝난 책'을 사람들에게 보여주는 곳이지, 관람객이 책의 내용을 그 자리에서 '지우개로 지우고 고쳐 쓸 수 있는 공책'이 아닙니다.

1) Vercel은 '읽기 전용(Read-Only)' 환경입니다
현재 이 포털은 메뉴의 설정값(사용/숨김 여부 등)을 .data/menuData.json 이라는 텍스트 파일에 적어서 저장하는 '로컬 파일 시스템' 방식을 사용합니다. 하지만 Vercel과 같은 클라우드 호스팅 서버는 보안 및 최적화를 위해, 배포된 서버 내부에 있는 파일을 실시간으로 덮어쓰거나 영구적으로 수정하는 행위를 원천적으로 차단하고 있습니다.

2) 저장을 눌러도 원래대로 돌아오는 현상의 원인
Vercel 링크에 접속하여 관리자 페이지에서 메뉴 스위치를 켜고 '저장' 버튼을 누르면, 시스템은 어떻게든 텍스트 파일을 수정해보려고 시도합니다. 하지만 Vercel 서버가 "여기는 파일 수정 금지 구역입니다!" 하고 튕겨내거나, 일회용 임시 공간에 잠깐 썼다가 곧바로 삭제해버립니다. 이 때문에 화면이 새로고침 되면, Vercel이 맨 처음 배포될 때 가지고 있던 원본(과거) 데이터를 그대로 다시 불러와서 보여주게 되는 것입니다.

3) 올바른 메뉴 관리 방법
이러한 한계 때문에, 포털의 메뉴를 추가하거나 숨기는 등의 '관리자 기능'은 반드시 로컬(내 컴퓨터) 환경에서 바탕화면의 실행기.bat를 통해 띄운 창에서만 진행하셔야 합니다. 개인 PC에서는 파일 덮어쓰기가 자유로우므로 저장이 정상적으로 처리됩니다. 로컬에서 수정을 완료하신 후, 터미널(CMD) 창에서 vercel --prod 명령어를 입력하여 최신 장부를 Vercel에 통째로 덮어쓰기(배포) 하셔야만 모든 사용자에게 변경된 메뉴가 완벽하게 노출됩니다.

8. Supabase 클라우드 DB 도입 (실시간 동기화로 Vercel 배포 문제 해결)

문제 해결: 과거 7번 항목의 한계(로컬 저장 후 매번 Vercel 배포 필요)를 완벽하게 극복하기 위해, 파일 저장 방식 대신 Supabase (서버리스 클라우드 DB)를 전격 도입했습니다.

1) 서버리스 DB 환경 구축
과거에는 메뉴 설정 데이터가 로컬 컴퓨터의 menuData.json 텍스트 파일에 머물러 있었으나, 이제는 24시간 안전하게 관리되는 Supabase 클라우드 데이터베이스에 직접 저장됩니다.

2) 별도의 배포 절차 없이 100% 실시간 반영
가장 큰 변화는 '실시간 동기화'입니다. 관리자 페이지에서 메뉴를 수정, 추가, 삭제하거나 순서를 변경한 뒤 [저장] 버튼을 누르면, 이 데이터가 클라우드 DB로 즉시 전송됩니다. 더 이상 검은 화면(CMD)을 열고 귀찮게 vercel --prod 명령어를 쳐서 배포(업데이트)할 필요가 없습니다. 저장 즉시 전 세계 어디서든 포털에 접속한 모든 사용자의 화면에 변경된 사항이 자동으로 반영됩니다.

9. Supabase Storage 도입 (파일 시스템의 클라우드화)

배경 및 목적: 데이터베이스(Supabase) 해결 이후, HTML 문서나 파이썬 스크립트 등 '파일' 자체에 대한 동기화 문제도 추가로 개선했습니다. 신규 메뉴가 등록되어 파일을 추가하거나 기존 파일의 소스코드를 변경했을 때, 여전히 로컬에서 수정 후 배포를 해야만 Vercel 환경에 반영되는 불편함이 남아 있었습니다.

1) 로컬 파일을 클라우드로 전면 이관
이 문제를 근본적으로 해결하기 위해 Supabase Storage (클라우드 파일 저장소)를 도입했습니다. 기존에 로컬 하드디스크(Project files/)에 담겨있던 모든 파일들을 클라우드 스토리지 공간으로 완전히 이관시켰습니다.

2) 완전한 서버리스 환경의 완성
결과적으로 포털의 모든 메뉴는 이제 로컬 PC가 아닌 Supabase 스토리지의 클라우드 파일만을 다이렉트로 바라보도록 구조가 개편되었습니다. 관리자 페이지에서 파일을 업로드하거나 수정하는 즉시 클라우드 스토리지에 바로 덮어쓰기가 진행되므로, 텍스트 데이터뿐만 아니라 '파일 데이터' 역시 단 한 번의 배포 과정 없이 100% 실시간으로 반영됩니다.

10. 사내 로그인 인증 구조 (왜 Vercel이 아닌 Supabase에만 보안키를 넣었을까?)

핵심 요약: Microsoft 사내 계정 연동을 위해 필요한 핵심 열쇠(테넌트 ID, 보안 비밀번호, App ID 등)를 화면을 보여주는 Vercel이 아니라, 뒷단의 Supabase(데이터베이스/인증 서버)에만 보관하도록 설계했습니다.

Vercel (건물의 안내데스크)

비유하자면 Vercel은 누구나 오갈 수 있는 '건물 1층의 투명한 안내데스크'입니다. 화면(HTML, JS)을 방문자에게 매우 빠르게 배포하는 데 특화되어 있습니다. 하지만 코드가 방문자의 인터넷 브라우저(크롬 등)에 그대로 노출될 수 있기 때문에, 이곳에 회사 전체의 보안 출입 명부를 통째로 맡기거나 마스터 키(보안 비밀번호)를 두는 것은 대단히 위험합니다.

Supabase (지하의 철통 금고실)

반면 Supabase는 외부에서 절대 코드를 뜯어볼 수 없는 '지하의 철통 금고실(백엔드 서버)'입니다. 그래서 Microsoft 시스템과 소통하기 위한 진짜 마스터 키(테넌트 ID, 보안 Secret ID 등)를 오직 Supabase 금고 안에만 안전하게 설정해 두었습니다.

실제 로그인 진행 과정 (안전한 역할 분담)

1. 사용자가 Vercel 안내데스크에서 [로그인 버튼]을 누릅니다.
2. 안내데스크는 직접 검사를 하지 않고, 사용자를 지하의 Supabase 금고실로 내려보냅니다.
3. Supabase는 금고 안에 숨겨둔 마스터 키를 꺼내어 Microsoft 본사(Azure AD)에 "이 사람 우리 직원 맞나요?" 라고 은밀하게 확인(인증)을 받습니다.
4. Microsoft가 "맞다"고 승인하면, Supabase는 사용자에게 안전한 '임시 출입증(토큰)'만 발급해 줍니다.
5. 사용자는 그 출입증만 들고 Vercel로 돌아와 포털을 자유롭게 이용합니다.

결과적으로: 이 방식을 통해 회사의 민감한 보안 키가 인터넷이나 사용자 브라우저에 단 한 글자도 유출되지 않으며, 관리자는 파편화된 Vercel 환경을 신경 쓸 필요 없이 오직 중앙 금고인 Supabase 단 한 곳에서만 보안 키를 간편하고 강력하게 통제할 수 있습니다.

11. 파이썬(.bat) 직접 실행 불가 문제와 현재의 대안 (남은 숙제)

현재의 한계: DB와 스토리지를 모두 클라우드화 했음에도 불구하고, 웹 브라우저의 강력한 보안 정책 때문에 Vercel 환경에서 로컬 PC의 파이썬(.bat) 프로그램을 마우스 클릭 한 번으로 직접 가동할 수 없다는 문제가 여전히 숙제로 남았습니다.

대안 1) 실행 파일(.bat) 수동 다운로드 방식
기존의 파이썬 데스크톱 UI를 그대로 유지하고 싶은 경우를 위해, 웹 화면에서 .bat 파일을 다운로드할 수 있는 기능을 제공했습니다. 사용자가 파일을 내려받아 직접 더블클릭하면 내부적으로 파이썬 코드를 백그라운드에서 실행하여 익숙한 프로그램 창이 열리도록 연동했습니다.

대안 2) 파이썬 로직의 웹 페이지 이관 (실험적 단계)
궁극적으로 다운로드조차 필요 없는 웹 환경을 구축하기 위해, 기존 파이썬 소스의 내용(PDF 파싱, 엑셀 생성)을 분석하여 웹 브라우저 상에서 즉시 동작하도록 별도의 웹 UI(Next.js API 연동)를 직접 구현해 보았습니다. 기능적인 구동은 성공했으나, 아직 UI의 심미적인 디테일 개선과 추가적인 기능 검증이 필요하여 이 부분은 계속해서 풀어나가야 할 과제(숙제)로 남겨두었습니다.

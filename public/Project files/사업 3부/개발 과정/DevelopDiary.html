바이브 코딩을 활용한 갤러리 웹사이트 구축 여정

마우스 휠이나 방향키(↓)로 내려보세요!

아무런 배경 없이 Gemini에서 시작

  • 초기에는 별도의 사전 지식이나 치밀한 계획 없이, 순수하게 바이브 코딩만으로 화면 구상을 시작했습니다.
  • Gemini의 Canvas 기능을 활용하여 초기 프로토타입 제작을 시도했습니다.
  • 전체적인 UI가 신속하게 생성되는 점은 무척 인상적이었습니다.
  • 하지만 프로젝트가 진행될수록 기능 추가 및 유지보수를 위한 확장성 측면에서 명확한 한계점을 체감하게 되었습니다.

Antigravity로 환승

  • 근본적인 한계를 극복하기 위해 프로젝트를 처음부터 다시 재구축하기로 결정했습니다.
  • 새로운 개발 도구로 Antigravity를 도입하였으며, 기본적인 기술 스펙을 지정하여 환경 구성을 요청했습니다.
  • AI가 프로젝트에 필요한 디렉토리 구조와 초기 파일들을 자체적으로 생성하는 것을 확인할 수 있었습니다.
  • 안정적으로 구축된 기반 위에서 Antigravity 툴을 적극 활용하여 본격적인 갤러리 사이트 개발에 착수했습니다.

기능 확장에 대한 니즈 (욕심의 시작)

  • 기존 자료(HTML, URL, Python 스크립트 등)를 업로드하고 각각의 메뉴를 통해 열람할 수 있는 통합 갤러리 화면을 완성했습니다.
  • 기본적인 뷰어(Viewer) 기능이 갖춰지자, 추가적인 시스템 관리에 대한 니즈가 발생했습니다.
  • "이 요소들을 관리자 페이지에서 중앙 제어할 수 있지 않을까?" 라는 가설을 세웠습니다.
  • 이러한 아이디어를 바탕으로 메뉴 생성/수정, 통계 확인, 그리고 화면 부연 설명을 위한 말풍선 기능 등을 포함한 관리자 시스템을 구축했습니다.

백엔드의 필요성

  • 다양한 데이터와 설정값들을 체계적으로 관리하기 위해서는 영구적인 저장 공간이 필수적이었습니다.
  • 휘발성 메모리에만 의존하는 방식으로는 앞서 기획한 관리자 기능들을 온전히 구현할 수 없었습니다.
  • 이에 따라 바이브 코딩을 통해 별도의 데이터 저장 로직 구현을 요청했습니다.
  • 결과적으로 물리적 공간(로컬 PC)을 활용하여 .json 파일 형태로 데이터를 안전하게 보존하는 방식을 채택했습니다.

Vercel 배포와 데이터 불일치

  • 클라이언트 화면 및 로컬 저장소 구축을 완료한 후, 서비스를 라이브로 제공하기 위해 Vercel 플랫폼에 연동했습니다.
  • 일부 설정 상의 시행착오가 있었으나, 결과적으로 Vercel 환경으로의 배포에 성공했습니다.
  • 하지만 배포 직후, 예상치 못한 '데이터 동기화 문제'에 직면하게 되었습니다.
  • 로컬 환경과 Vercel 환경의 데이터가 상호 공유되지 않아, 접속 환경에 따라 완전히 다른 데이터를 렌더링하는 치명적인 이슈가 발생한 것입니다.

Supabase를 통한 데이터 동기화

  • 웹 서비스에서 데이터의 무결성과 일관성은 핵심 요소이므로, 단일화된 중앙 데이터베이스(DB)의 도입이 시급했습니다.
  • 이를 해결하기 위한 방안으로 서버리스 데이터베이스인 Supabase를 채택했습니다.
  • Supabase와 Antigravity 환경을 연동하고, 필요한 테이블을 설계하여 클라이언트 화면과 성공적으로 매핑했습니다.
  • 마침내 저장 공간이 하나로 통합되었으며, 어떠한 접속 환경에서도 완벽하게 동일한 데이터를 보장할 수 있게 되었습니다.

파일 스토리지의 필요성 (Vercel Storage)

  • Supabase를 통한 데이터 관리뿐만 아니라, HTML 문서나 스크립트 등 물리적인 파일 관리의 필요성도 체감하게 되었습니다.
  • 주된 목적은 원활한 '메뉴 관리'였습니다. 신규 메뉴를 등록하거나 기존 파일을 수정할 때마다 로컬에서 작업 후 매번 재배포를 거쳐야 하는 번거로움이 있었습니다.
  • 이러한 파일 동기화 역시 Vercel 환경에서 실시간으로 처리하기 위해 Vercel Storage (클라우드 파일 저장소)를 전격 도입했습니다.
  • 로컬에 있던 모든 소스 파일들을 클라우드 스토리지로 완전히 이관하고, 포털의 모든 메뉴가 로컬이 아닌 Vercel 스토리지만을 다이렉트로 바라보도록 개편했습니다.
  • 그 결과, 로컬이든 Vercel 환경이든 어디서 수정하더라도 별도의 배포 없이 100% 즉각적인 실시간 동기화가 가능해졌습니다!

Vercel 호스팅의 한계점

  • Vercel은 훌륭한 프론트엔드 호스팅 환경을 제공하지만, 로컬 환경처럼 파이썬(.py)이나 배치 파일(.bat)을 직접 실행하여 결과를 반환하는 아키텍처는 지원하지 않음을 확인했습니다.
  • Github와의 연동을 통해 정적 파일 및 버전을 관리하는 우회적인 방법도 고려해 보았습니다.
  • 그러나 해당 방식은 실시간 로딩이 불가능하며, 변경 사항 반영에 1~2분의 지연(Delay)이 발생하여 즉각적인 피드백이 어렵다는 단점이 있습니다.
  • 이러한 실행 환경의 제약은 향후 서비스 고도화를 위해 추가적으로 연구해야 할 과제로 남았습니다.

프로젝트 인사이트 (배운 점)

  • 이번 프로젝트를 통해, 본격적인 개발에 앞서 명확한 아키텍처 설계가 선행되어야 함을 다시 한번 실감했습니다.
  • 단순한 로컬 구동용인지, 혹은 클라우드 환경에서 다수의 사용자를 대상으로 서비스할 것인지 명확한 목적 설정이 필요합니다.
  • 백엔드 시스템 도입 시 데이터의 성격에 맞춰 저장 방식(메모리, 로컬 파일, 온라인 DB 등)을 사전에 확정해야 합니다.
  • 향후 프로젝트에서는 충분한 요구사항 분석과 사전 설계를 바탕으로, 체계적이고 단계적인 개발 방법론을 적용할 계획입니다.
